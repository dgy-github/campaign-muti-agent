campaign · command / runtime / rehearsal

不是 Demo，是一套可运行的多 Agent 战役框架

campaign 把多 Agent 从“几个角色聊天”推进到工程系统：事件溯源做状态底座，Coordinator / Executor / Retriever / Reviewer / Reserve 分工协作，PolicyGate 统一治理，A2A 传输可替换，Eval + Chaos 把上线前演练变成门禁。

English summary: campaign is a runnable multi-agent command, runtime, and rehearsal framework. It treats agents as a governed production system: event-sourced state, explicit A2A messages, policy gates, durable HITL resume, RAG memory, observability, and chaos/eval release gates.

01项目定位

一套可以拿来讲架构、跑 demo、做面试项目背书的 multi-agent reference implementation。

What

指挥-运行-演练框架

不是单个 agent，而是一个能把任务拆解、路由、执行、审查、治理、回放、演练串起来的框架骨架。

Why

把 Agent 做成可控系统

多 Agent 最怕失控、不可复盘、成本爆炸和责任不清。campaign 用事件源、治理闸和独立 Reviewer 把这些问题变成工程约束。

Proof

172 测试不是摆设

测试覆盖 A2A、HTTP hardening、SSE、HITL、DAG、预算、RAG、记忆、混沌演练、capstone 集成链路。

面试一句话：我把多 Agent 拆成“指挥层、运行时、演练层、横切治理层”，状态全部由事件派生，所有外部调用先过 PolicyGate，最后用 Eval/Chaos 做上线门禁。这比“几个 agent 互相 chat”更接近真实工程。

02分层架构

四层主链路，三条横切能力，保证每个角色知道自己该做什么、不该碰什么。

03一次任务怎么跑

从用户目标到最终验收，所有关键点都有事件、有边界、有失败处理。

1. run.started分配 run_id，写入 EventLog，建立本次执行边界。
2. decomposeCoordinator 把 ExecutionOrder 拆成可执行 Task。
3. routeSkillRegistry + ROI Router 选择最合适 agent。
4. gatePolicyGate 检查预算、权限、数据出域和注入风险。
5. dispatchTask 封装成 Message，经 Transport 送达 worker。
6. reviewReviewer 独立验收，写 score 与 done/failed 事件。
关键点：worker 不被 runtime 直接裸调，而是走 Message.request -> Transport.send -> on_message。这让本地 agent、远程 agent、SSE 流式 agent 可以共享同一套执行语义。
成功路径
task.assigned -> executor.output -> review.done(score) -> task.done -> run.completed
失败路径
task.failed 会拉低 health，触发 breaker / brake / mobilization；如果是 HITL，写 input_required 后等待 Runtime.resume()。
并发路径
set_concurrency(n) 使用 Semaphore + gather；DAG 任务通过 depends_on 拓扑分层，避免依赖未完成就乱跑。

04事件溯源底座

状态不是“存在内存里的当前值”，而是由事件回放派生出来的态势。

EventLog = append-only 作战日志

SQLite 存储，seq 单调递增，所有 run、task、message、review、brake、governance 事件都只追加不覆盖。这样天然支持审计、回放、跨进程恢复和故障后复盘。

run_id 隔离 = 多战役不串线

derive_state(events, run_id) 只折叠当前 run 的事件，多次运行、远程协作、换设备 resume 都不会把任务态混到一起。

事件意义
run.started / run.completed一次执行令的开始与结束，所有后续事件挂到同一个 run_id。
task.assigned / task.done / task.failed任务生命周期，不靠“猜当前状态”，靠事件回放得出。
a2a.messageA2A 消息审计，记录 from/to/kind/correlation_id。
governance.violation越权、超预算、敏感出域等治理违规，进入同一条审计总线。
brake.signal / mobilization.*熔断、协同制动、减员动员，这些韧性动作也可复盘。

05A2A 协议与远程协作

本地 agent 和远程 agent 共享同一个协议信封，Transport 可替换。

Protocol

Message / Part / AgentCard

Message 管 from/to、run_id、task_id、correlation_id 和 kind；Part 标记 text/data 与 untrusted；AgentCard 用于能力发现。

Transport

InProcess + HTTP + SSE

InProcess 用于本地路由；HTTP JSON-RPC 支持重试、退避、Bearer 鉴权和 message_id 幂等；SSE 用于流式状态事件。

能力实现方式
远程调用HttpJsonRpcTransport 把 Message 序列化为 JSON-RPC 2.0，请求失败转成 error Message，不让异常炸穿 runtime。
幂等去重server 按 message_id 缓存响应，重试不会重复执行 worker。
能力发现agent/cards -> discover() / Runtime.discover_remote() 自动注册远程 agent。
分布式 MVP节点 A 通过 RemoteAgentProxy 派活给节点 B，共享 SQLite EventLog，统一 derive_state。
诚实边界：当前分布式是 MVP：HTTP 传输 + 共享事件库 + 远程代理已经跑通，但预算、health、熔断、mobilizer leader 仍是每进程维度，生产级一致性还需要单独治理。

06治理、韧性、可观测

多 Agent 真正难的不是“会干活”，而是“失控时能刹住、能解释、能恢复”。

Governance

PolicyGate 统一执行闸

预算、数据出域、越权、注入扫描统一过闸。runtime 每个 task 和 LLMClient 每次调用都要经过 gate，而不是只在入口做一次样子。

Resilience

熔断 + 制动 + 动员减员

失败降低 health，breaker 进入 open 状态，brake 做协同背压；mobilizer 根据真实健康信号做三层级减员。

Observability

Tracer span / meter / score

每次 agent 调用、review 分数、token 计量都进入可观测通道，可对接 Langfuse / OpenTelemetry。

07记忆与检索

短期状态、中期会话、长期知识分开，避免把“记忆”做成一个黑盒垃圾桶。

层级组件作用
短期ScratchpadMemory本轮/近期执行用的临时上下文。
中期SessionMemory按 run_id 读取 EventLog，恢复同一次战役上下文。
长期SqliteKnowledgeStoreTF-IDF RAG 知识库，保留神经 embedding 接口缝。
接入enable_memory() / set_knowledge_store()opt-in 接入 runtime 主链，不破坏最小可运行路径。
安全处理：检索结果标记 untrusted，敏感内容做本地隔离；外部内容进入 agent 之前先被当作不可信数据，而不是系统指令。
